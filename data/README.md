# 데이터 디렉터리 안내

이 디렉터리는 광주 웨딩 박람회 수집 결과와 운영 상태 파일을 저장하는 곳입니다. 이 프로젝트에서 가장 먼저 확인해야 하는 운영 산출물도 이 디렉터리에 있습니다.

현재 기준으로 이 디렉터리의 파일은 “임시 참고용”이 아니라 실제 운영 상태를 반영하는 파일입니다. 따라서 오래된 설명이나 과거 검증 점수를 적어 두는 방식보다, 현재 데이터 구조와 확인 방법을 명확히 적는 것이 더 중요합니다.

## 1. 포함된 파일

### `gwangju_wedding_expos.csv`

현재 최종 canonical 결과를 저장하는 CSV 파일입니다.

이 파일은 다음 특징을 가집니다.

- 대상 범위는 오늘부터 향후 3개월의 광주 웨딩 박람회
- 이미 시작했더라도 아직 끝나지 않은 진행 중 행사 포함
- 여러 출처에서 들어온 중복 행사는 하나의 최종 레코드로 병합
- 실행이 끝날 때마다 최신 스냅샷으로 다시 생성

즉 이 파일은 append 로그가 아니라 “현재 운영 결과의 정답 파일”에 가깝습니다.

### `wedding_expos.db`

SQLite 데이터베이스 파일입니다. 대시보드와 내부 조회에서 사용됩니다.

현재 운영 정책상 DB도 CSV와 같은 최종 스냅샷을 저장합니다. 따라서 CSV와 DB는 같은 시점의 데이터를 본다고 생각하면 됩니다.

### `source_health.json`

소스별 헬스 상태 파일입니다. 각 수집 소스의 최근 성공 여부, 연속 실패 수, 연속 0건 수, 마지막 결과 수 등이 저장됩니다.

이 파일은 서킷 브레이커의 기반 데이터이기도 합니다.

### `logs/`

최신 실행 리포트와 로그 파일이 들어가는 디렉터리입니다.

특히 아래 파일이 중요합니다.

- `logs/latest_source_health_report.json`

이 파일에는 이번 실행의 요약이 들어 있습니다.

- `raw_count`
- `parsed_count`
- `final_valid_count`
- `coverage_reference_count`
- `coverage_matched_count`
- `coverage_missing_count`
- `zero_result_sources`
- `critical_zero_result_sources`
- `skipped_sources`

## 2. 현재 CSV 컬럼

`gwangju_wedding_expos.csv`는 현재 아래 컬럼을 가집니다.

- `name`
- `start_date`
- `end_date`
- `operating_hours`
- `location`
- `organizer`
- `contact`
- `source_url`
- `description`
- `region`
- `source`

이 컬럼 구성은 대시보드, DB, 후속 발행 작업에서 공통으로 사용됩니다.

## 3. 데이터 해석 기준

이 디렉터리의 데이터는 아래 기준으로 해석해야 합니다.

### 날짜 범위

- 오늘 이후 시작하는 행사 포함
- 오늘 이전에 시작했더라도 아직 종료되지 않은 행사 포함
- 오늘부터 3개월을 넘는 미래 일정은 제외

즉 “진행 중 행사”가 빠져 있으면 비정상입니다.

### 지역 범위

- 광주광역시 행사 중심
- 전라도 또는 전국 소스에서 수집되더라도 실제 행사 장소가 광주인 경우 포함

### 중복 정리

동일 행사로 판단되면 출처가 달라도 하나로 정리됩니다. 이때 더 정확한 장소, 더 믿을 수 있는 출처, 더 많은 필드가 채워진 레코드가 우선 선택됩니다.

## 4. 정상 상태를 판단하는 방법

이 디렉터리만 보고도 아래 항목을 점검할 수 있습니다.

### 4.1 CSV 확인

다음을 확인합니다.

- 오늘 기준 유효한 광주 행사들이 실제로 들어 있는지
- 진행 중 행사도 포함돼 있는지
- 장소가 광주 상세 주소로 정규화됐는지
- 테스트용 더미 데이터가 섞여 있지 않은지

### 4.2 헬스 리포트 확인

`logs/latest_source_health_report.json`에서 다음을 확인합니다.

- `final_valid_count > 0`
- `coverage_reference_count > 0`
- `coverage_missing_count == 0`

이 세 가지가 맞아야 현재 공개 참조 소스 기준으로 누락 없이 수집되고 있다고 볼 수 있습니다.

## 5. CSV가 이상할 때 점검 순서

CSV가 비어 있거나, 예전 데이터가 남아 있거나, 테스트 데이터처럼 보이면 아래 순서로 확인합니다.

1. `python3 main.py --dry-run --skip-github --skip-tistory --skip-notify`
2. `data/logs/latest_source_health_report.json` 확인
3. `python3 main.py --skip-github --skip-tistory --skip-notify`
4. CSV 재확인
5. 필요 시 대시보드와 `source_health.json` 확인

현재 코드는 테스트가 운영 CSV를 직접 덮어쓰지 않도록 분리돼 있어야 정상입니다.

## 6. 자주 묻는 질문

### Q. CSV는 누적 저장인가요?

아닙니다. 최신 canonical 결과를 기준으로 매 실행마다 다시 생성하는 스냅샷 파일입니다.

### Q. DB와 CSV가 다를 수 있나요?

정상이라면 거의 같아야 합니다. 둘 다 같은 최종 결과를 기준으로 갱신됩니다.

### Q. 왜 행 수가 매번 달라질 수 있나요?

다음 이유 때문입니다.

- 오늘 날짜가 바뀌어 3개월 필터 결과가 달라짐
- 외부 소스가 새 행사나 종료된 행사를 반영함
- 중복 병합 결과가 달라짐
- 특정 소스가 일시적으로 0건 또는 실패 상태가 됨

### Q. 완벽한 데이터인가요?

기술적으로 완벽을 보장할 수는 없습니다. 다만 현재 공개 참조 소스 기준으로 누락이 없는지 자동 감사하며, 누락이 감지되면 운영 가드가 본실행을 막도록 구성돼 있습니다.
